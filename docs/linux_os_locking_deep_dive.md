# Linux/操作系统视角下的锁底层原理

本文从 CPU、缓存一致性、Linux 内核、线程调度、futex 和 C++11 原子操作几个层次解释锁为什么能工作，以及标准库锁和本项目自旋锁的底层差异。

本项目实际依赖的每个 C++11 原子操作的详细语义、可能的 CPU 指令映射和状态转换分析，见 [C++11 原子操作底层原理与实现](atomic_operations_deep_dive.md)。

## 1. 锁要解决什么问题

多线程程序的核心问题是：多个执行流同时访问同一份共享数据。

如果至少有一个线程写入共享数据，并且访问之间没有同步，就会产生数据竞争。数据竞争会导致：

- 读到旧值。
- 写入丢失。
- 观察到中间状态。
- 编译器或 CPU 重排后得到违反直觉的执行顺序。
- C++ 层面进入未定义行为。

锁的作用不是让 CPU “真的只运行一个线程”，而是通过一个共享状态变量建立进入临界区的规则：

```text
线程 A 想进临界区
    -> 原子地检查锁状态
    -> 如果空闲，把锁改成已占用
    -> 成功后访问共享数据
    -> 退出时释放锁

线程 B 想进临界区
    -> 如果发现锁已占用
    -> 等待、重试或睡眠
```

关键点是“检查锁状态”和“修改锁状态”必须是一个不可分割的原子操作。

## 2. 为什么普通变量不能实现锁

错误示例：

```cpp
bool locked = false;

void lock() {
    while (locked) {
    }
    locked = true;
}
```

这个实现有两个严重问题。

### 2.1 检查和修改不是原子的

可能发生这样的执行顺序：

```text
线程 A 读取 locked == false
线程 B 读取 locked == false
线程 A 写入 locked = true
线程 B 写入 locked = true
线程 A 和线程 B 同时进入临界区
```

两个线程都认为自己获得了锁，互斥性被破坏。

### 2.2 编译器和 CPU 可以重排

普通变量没有同步语义。编译器可能把读写缓存到寄存器，也可能调整指令顺序。CPU 也可能通过 store buffer、乱序执行等机制改变其他核心观察到的顺序。

锁必须使用原子操作和内存序来约束这些行为。

## 3. CPU 原子指令

现代 CPU 提供原子读改写指令，用来实现锁。

常见原子操作：

- test-and-set：读取旧值，并写入新值。
- compare-and-swap/CAS：如果当前值等于 expected，就改成 desired。
- fetch-add：读取旧值，并加上一个增量。
- exchange：读取旧值，并替换成新值。

在 C++11 中对应：

```cpp
std::atomic_flag::test_and_set()
std::atomic<T>::compare_exchange_weak()
std::atomic<T>::compare_exchange_strong()
std::atomic<T>::fetch_add()
std::atomic<T>::exchange()
```

### 3.1 x86 上的常见实现

x86/x64 上，原子操作常映射到带 `lock` 前缀的指令，例如：

```text
lock cmpxchg
lock xadd
xchg
```

这些指令会让目标内存地址对应的缓存行在多核之间完成独占修改。现代 CPU 不是简单地锁整条总线，而是依赖缓存一致性协议让某个核心获得该缓存行的独占所有权。

### 3.2 ARM 上的常见实现

ARM 上常见路径是 load-exclusive/store-exclusive：

```text
LDXR / STXR
LDAXR / STLXR
```

简化理解：

1. `LDXR` 读取一个地址，并标记当前核心正在独占观察它。
2. 其他核心如果修改了该地址，当前核心的独占标记失效。
3. `STXR` 只有在标记仍有效时才写入成功。

这正好可以实现 CAS 循环。

## 4. 缓存一致性与锁竞争

多核 CPU 每个核心都有自己的缓存。锁变量通常位于某个缓存行中。

当多个核心频繁修改同一个锁变量时，会出现缓存行来回迁移：

```text
Core 0 想修改锁变量 -> 获得缓存行独占权限
Core 1 想修改锁变量 -> 缓存行从 Core 0 迁移到 Core 1
Core 2 想修改锁变量 -> 缓存行再迁移到 Core 2
```

这叫缓存行争用。自旋锁在高竞争下慢，很多时候不是因为循环本身，而是因为所有线程都在争抢同一个缓存行。

### 4.1 test-and-set 自旋锁的问题

本项目 `Mutex` 使用的是教学上最直接的 test-and-set 自旋锁：

```cpp
while (locked_.test_and_set(std::memory_order_acquire)) {
    std::this_thread::yield();
}
```

优点：

- 实现简单。
- 无竞争时非常快。
- 非常适合讲解原子操作和内存序。

缺点：

- 每次失败都会尝试写锁变量。
- 写操作会导致缓存行独占迁移。
- 高竞争下缓存一致性流量很大。

生产级自旋锁常会优化成 test-test-and-set：

```text
先普通 load 观察锁是否空闲
只有看起来空闲时才执行 test_and_set
```

这样失败等待阶段主要是读共享缓存行，减少写争用。

## 5. 内存序与锁语义

锁除了保证“同一时刻只有一个线程进入临界区”，还必须保证“临界区内的数据修改对下一个持锁线程可见”。

典型模型：

```text
线程 A:
lock()
shared_data = 100;
unlock()

线程 B:
lock()
读取 shared_data
unlock()
```

如果 `unlock()` 没有 release 语义，`shared_data = 100` 可能在其他核心看来晚于锁释放。

如果 `lock()` 没有 acquire 语义，线程 B 获取锁后也可能看不到线程 A 的临界区写入。

因此本项目采用：

```cpp
lock 成功: memory_order_acquire
unlock: memory_order_release
```

### 5.1 acquire 的作用

acquire 作用于加锁成功的原子操作：

- 后面的普通读写不能被重排到 acquire 之前。
- 读取到 release 写入后，可以看到 release 之前的普通写入。

### 5.2 release 的作用

release 作用于解锁操作：

- 前面的普通读写不能被重排到 release 之后。
- 让后续 acquire 成功的线程看到临界区内的结果。

### 5.3 relaxed 为什么不够

`memory_order_relaxed` 只保证原子变量本身不会撕裂读写，不建立跨线程同步关系。

用 relaxed 可以保护锁状态变量不坏，但不能完整表达临界区内共享数据的可见性。因此锁实现通常不能只用 relaxed。

## 6. Linux 用户态锁的常见结构

Linux 上成熟的互斥锁一般不是单纯自旋，而是“用户态快速路径 + 内核态慢路径”。

典型结构：

```text
lock()
    -> 用户态 CAS 尝试获得锁
    -> 成功：直接进入临界区，不进内核
    -> 失败：通过 futex 进入内核等待

unlock()
    -> 用户态释放锁状态
    -> 如果发现有人等待，通过 futex 唤醒等待线程
```

这样设计的原因：

- 无竞争时避免系统调用，速度接近纯原子操作。
- 有竞争时让线程睡眠，避免长时间空转消耗 CPU。

## 7. futex 的核心思想

futex 是 fast userspace mutex 的缩写。它不是一个完整 mutex，而是 Linux 提供的“基于用户态整数地址的等待/唤醒机制”。

用户态维护一个整数状态，例如：

```text
0 = 未上锁
1 = 已上锁且无人等待
2 = 已上锁且可能有人等待
```

当竞争失败时，线程调用 futex wait：

```text
futex_wait(address, expected_value)
```

内核会做两件关键事：

1. 再次检查 `*address` 是否仍等于 expected value。
2. 如果仍相等，把当前线程挂到这个地址对应的等待队列并睡眠。

解锁线程调用 futex wake：

```text
futex_wake(address, wake_count)
```

内核从等待队列里唤醒一个或多个线程。

### 7.1 为什么 futex 要检查 expected value

这是为了避免丢失唤醒。

如果没有内核中的原子检查，可能发生：

```text
线程 A 准备睡眠
线程 B 解锁并唤醒
线程 A 还没真正进入等待队列，错过唤醒
线程 A 睡眠后再也没人叫醒
```

futex wait 把“检查值”和“入队睡眠”做成内核里的原子序列，避免这个问题。

## 8. Linux 内核调度与阻塞锁

当线程通过 futex、条件变量、IO 等机制阻塞时，Linux 内核会把线程从可运行队列移走。

简化状态：

```text
TASK_RUNNING
    -> 可以被调度运行

TASK_INTERRUPTIBLE / TASK_UNINTERRUPTIBLE
    -> 阻塞等待某个事件
    -> 不占用 CPU 时间片
```

被唤醒后，线程重新进入可运行队列，等待调度器分配 CPU。

这就是阻塞锁比自旋锁适合长临界区的原因：

- 自旋锁等待时线程仍在运行，占用 CPU。
- 阻塞锁等待时线程睡眠，把 CPU 让给其他任务。

## 9. 自旋锁在内核里的使用场景

Linux 内核也大量使用自旋锁，但使用场景非常谨慎。

适合自旋锁的场景：

- 临界区非常短。
- 不能睡眠的上下文，例如中断上下文。
- 持锁时间可以被严格控制。

不适合自旋锁的场景：

- 临界区可能执行 IO。
- 临界区可能阻塞。
- 临界区很长。
- 竞争线程数很多。

内核自旋锁还会涉及：

- 禁止抢占。
- 中断上下文保护。
- SMP 多核下的缓存一致性。
- debug 配置下的死锁检测。

本项目自旋锁是用户态教学实现，不包含这些内核级机制。

## 10. 读写锁的底层思想

读写锁利用“读读不互斥，读写互斥，写写互斥”的访问模式提高并发度。

核心状态通常至少包含：

```text
reader count
writer active
waiting writers
```

本项目把 reader count 和 writer active 合并到一个整数：

```text
state_ == -1: writer active
state_ == 0 : unlocked
state_ > 0  : reader count
```

写锁获取：

```text
只有 state_ == 0 时才能 CAS 成 -1
```

读锁获取：

```text
只有 state_ >= 0 时才能把 state_ 加 1
```

这个状态机保证：

- 写线程持锁时，读线程无法进入。
- 有读线程时，写线程无法进入。
- 没有写线程时，多个读线程可以同时进入。

## 11. 公平性与饥饿问题

读写锁如果不控制读线程进入，写线程可能饥饿。

典型问题：

```text
已有读线程 R1
写线程 W 开始等待
新的读线程 R2、R3、R4 不断进入
读者数量一直不归零
W 长时间拿不到写锁
```

本项目使用 `waiting_writers_` 做简化写优先：

```text
写线程等待时 waiting_writers_ > 0
新的读线程看到 waiting_writers_ > 0 后暂缓进入
已有读线程退出后，写线程更容易获得锁
```

这不是严格公平队列，但能解释写优先策略的核心思想。

严格公平读写锁通常需要：

- 等待队列。
- 每个等待线程的节点。
- 唤醒策略。
- 读者批次管理。
- 防止新读者插队。

这些机制一般需要条件变量、futex 或平台等待原语配合。

## 12. 标准库 std::mutex 与本项目锁的核心差异

标准库 `std::mutex` 的成熟实现通常是混合锁：

```text
无竞争:
    用户态 CAS 成功

短暂竞争:
    可能自旋几次

持续竞争:
    futex/系统等待队列睡眠
```

本项目锁：

```text
无竞争:
    用户态原子操作成功

持续竞争:
    yield 后继续用户态重试
```

因此：

- 本项目更适合展示原理。
- 标准库锁更适合生产。
- 本项目能清楚看到状态机和原子操作。
- 标准库实现隐藏了更多 OS 细节，但处理了性能、调度和公平性问题。

## 13. 面试可讲的完整链路

可以按这个顺序讲：

1. 多线程共享数据会产生数据竞争。
2. 普通 bool 无法实现锁，因为检查和修改不是原子的。
3. CPU 提供 test-and-set、CAS、fetch-add 等原子指令。
4. C++11 用 `std::atomic` 抽象这些原子能力。
5. 加锁用 acquire，解锁用 release，保证临界区可见性。
6. 互斥锁可以用一个原子标志实现。
7. 读写锁可以用一个原子整数状态机实现。
8. 自旋锁适合短临界区，长等待会浪费 CPU。
9. Linux 标准锁通常用 futex 做“用户态快路径 + 内核态睡眠慢路径”。
10. 生产锁还要处理公平性、饥饿、优先级反转、超时和诊断。
